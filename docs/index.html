Building a Distributed LLM Inference System from Scratch

A six-phase deep dive into production-grade LLM serving — from a single FastAPI endpoint to a fully observable, horizontally scalable inference platform.

📅 March 2026 ⏱ 20 min read 🔧 FastAPI · Redis · vLLM · OpenTelemetry · Prometheus

Why This Exists

Most ML engineers know how to train models. Very few know how to serve them at scale. The gap between a Jupyter notebook and a production inference system is enormous — it involves queuing, distributed tracing, GPU observability, and container orchestration.

This project closes that gap by building a real distributed inference service in six phases, each adding a production concept on top of the last. The model used is TinyLlama/TinyLlama-1.1B-Chat-v1.0 running on vLLM — small enough to run on a 6GB GPU, real enough to expose every infrastructure problem.

The key insight: the API gateway never waits for inference. It pushes a job to Redis and returns a request_id in milliseconds. Workers pull from the queue independently — add more workers, get more throughput.

Phase 1 — The Model Server

The starting point is a single FastAPI app that wraps vLLM's OpenAI-compatible API. The critical detail: TinyLlama is a chat model, so raw prompts produce incoherent output. The fix is wrapping every prompt in the model's chat template: 

formatted = (
    f"<|system|>\nYou are a helpful assistant.</s>\n"
    f"<|user|>\n{prompt}</s>\n"
    f"<|assistant|>\n"
)

Without this, "Explain transformers" produces content about fashion brands. With it, you get actual ML explanations. This is the most common mistake when deploying instruction-tuned models.

Phase 2 — Decoupling with Redis

A direct API-to-GPU pipeline has a fatal flaw: when the GPU is busy, requests queue inside the web server, blocking threads. Under load, the server becomes unresponsive.

The fix is a queue. The API returns immediately with a job ID. Workers pull from Redis using blpop (blocking pop — no polling loop needed). Clients poll /result/{id} until the response is ready. 

# API — returns in ~5ms
POST /generate?prompt=...
→ {"status": "queued", "request_id": "abc-123"}

# Client polls
GET /result/abc-123
→ {"status": "processing"}   # GPU still working
→ {"choices": [...]}         # done

Scaling insight: running three workers instead of one triples throughput with zero code changes. Jobs distribute automatically because blpop is atomic — only one worker claims each job.

Phase 3 — Distributed Tracing

Logs tell you what happened. Traces tell you where time was spent across services. Each request produces a tree of spans: 

api_request                          3ms
worker_process                      │
  queue_wait_ms = 41ms              │
  └── model_inference             118ms
                              ─────────
                              total: 162ms

This immediately shows the GPU is the bottleneck, not the queue or network. OpenTelemetry is used instead of the deprecated Jaeger Thrift exporter — OTLP is the modern wire protocol, compatible with Jaeger, Tempo, and every major backend.

Phase 4 & 5 — Metrics and GPU Observability

Tracing answers "why was request X slow?" Metrics answer "is the system healthy overall?"

Prometheus metrics exposed:

  • inference_requests_total — counter, tracks volume
  • inference_queue_wait_seconds — histogram, detects congestion
  • inference_latency_seconds — histogram, GPU speed
  • inference_tokens_per_second — histogram, generation throughput
  • gpu_utilization_percent — gauge, from nvidia-smi
  • gpu_memory_used_mb — gauge, VRAM tracking

GPU metrics are polled every 15 seconds in a background daemon thread inside the worker, keeping the inference loop unblocked.

What This Teaches

These six phases cover the core skills of ML infrastructure engineering:

  • Producer-consumer systems — Redis queue, async job dispatch
  • Distributed tracing — OpenTelemetry, span propagation, OTLP
  • Metrics engineering — counters, histograms, PromQL, Grafana
  • GPU observability — utilization, VRAM, token throughput
  • Container orchestration — Docker Compose, horizontal scaling
  • LLM serving specifics — chat templates, KV cache, continuous batching

These are exactly the skills worked on at companies building LLM infrastructure — NVIDIA, OpenAI, Databricks, and any team running models in production.

Running It

# Install
python -m venv venv && source venv/bin/activate
pip install fastapi uvicorn redis requests \
  opentelemetry-api opentelemetry-sdk \
  opentelemetry-exporter-otlp-proto-grpc \
  prometheus-client vllm

# Start vLLM (requires GPU)
python -m vllm.entrypoints.openai.api_server \
  --model TinyLlama/TinyLlama-1.1B-Chat-v1.0 \
  --port 8001 --gpu-memory-utilization 0.7 --max-num-seqs 32

# Start infrastructure
redis-server
docker run -d -p 16686:16686 -p 4317:4317 docker.io/jaegertracing/all-in-one

# Start API + worker
uvicorn src.inference_systems.api.server:app --port 8000
python src/inference_systems/workers/worker.py

# Benchmark
python scripts/benchmark.py --total 20 --concurrency 4

Or run everything at once: docker compose up -d (vLLM excluded — GPU passthrough varies by runtime).
